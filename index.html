Caveats

n = 1. A single bear clip; no statistical claim. The metric gaps shown are illustrative, not statistically significant.

Sora is not strict-frame I2V. Sora's actual frame[0] differs from the uploaded input_reference by 19% pixel-wise — the bear is in a different pose. input_reference is a soft scene/style anchor per the OpenAI cookbook ("anchor for the first frame"), not a literal first frame. DaS's --repaint mode mirrors this behaviour by regenerating frame 0 via depth-anything-v2 + FLUX.1-Depth-dev.

Prompt iteration matters. The legacy suffix ("only small and smooth movement") let Sora dolly; the new suffix ("keep the camera fairly static") locks the camera, raising Sora's Tem-Con +0.017 and Subj-Cons +0.011. DaS+repaint also uses the new suffix.

Sora pacing. Sora generated 4 s of bear walking; we trimmed to the first 2.04 s (round(D_target × sora_fps)). The trimmed action reflects Sora's pacing across the full 4 s — if motion is back-loaded we capture only setup motion. Eyeball the hero clip to confirm.

Sora 2 is 30 fps, not 24 as some docs claim. We probe the actual fps from the downloaded mp4 via cv2 and use it for the trim count.
