Best generative score per metric highlighted in green. n = 1 — single clip, no statistical claim.

Split winner. DaS leads on Tem-Con (matches GT — the track conditioning keeps frame-to-frame motion smooth); Sora 2 leads on Subj-Cons and exceeds GT (0.9659 vs 0.9626) because the locked-camera prompt ("keep the camera fairly static") removes the natural parallax/shake that lowers GT's per-frame DINO consistency. With an earlier suffix ("small and smooth movement") Sora scored only 0.9551 / 0.9573 (Subj-Cons / Tem-Con) — the strict camera lock improved both by >0.01.

Caveats

n = 1. A single bear clip; no statistical claim. The metric gaps shown are illustrative, not statistically significant.

Sora is not strict-frame I2V. Sora's actual frame[0] differs from the uploaded input_reference by 19% pixel-wise — the bear is in a different pose. input_reference is a soft scene/style anchor per the OpenAI cookbook ("anchor for the first frame"), not a literal first frame. DaS's demo.py --repaint true flag does similar regeneration via FLUX.1-Depth-dev; we tested it and the redrawn frame felt too synthetic, so we stayed with the literal-frame DaS path here.

Prompt suffix asymmetry. DaS uses the original "only small and smooth movement" suffix (the existing 2026-04-28 smoke output); Sora was rerun with the stricter "keep the camera fairly static" suffix because Sora interpreted the original wording as license to dolly. With the original suffix Sora scored 0.9551 / 0.9573 (Subj-Cons / Tem-Con); the strict camera lock improved both by >0.01.

Sora pacing. Sora generated 4 s of bear walking; we trimmed to the first 2.04 s (round(D_target × sora_fps)). The trimmed action reflects Sora's pacing across the full 4 s — if motion is back-loaded we capture only setup motion. Eyeball the hero clip to confirm.

Sora 2 is 30 fps, not 24 as some docs claim. We probe the actual fps from the downloaded mp4 via cv2 and use it for the trim count.
